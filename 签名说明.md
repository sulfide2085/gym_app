# Android 签名说明

## 为什么会提示签名不符

Android 允许覆盖安装更新包，但要求下面两个条件同时成立：

1. 包名相同
2. 新旧 APK 使用同一把签名证书

你现在遇到的“签名不符，不让更新”，通常是下面几种情况之一：

- 旧 App 是用以前的一把 keystore 签的
- 新 App 是 `debug` 包，旧 App 是 `release` 包
- 新 App 是未签名包，或者换了一把新的 keystore

## 关键结论

- 能持续更新同一个 App 的前提，不是“每次都重新签名”。
- 而是“每次都用同一把 keystore 和同一个 key alias 签名”。
- 这把 keystore 一旦丢失，就无法继续覆盖更新以前发出去的安装包。

## 当前项目的签名方案

当前项目已经支持从仓库根目录读取本地签名配置：

- 配置文件：`keystore.properties`
- 示例文件：`keystore.properties.example`
- keystore 文件建议放在：`keystore/`

这些文件不会提交到 Git。

## 推荐入口

当前推荐使用：

- `release.py`：主逻辑
- `release.bat`：Windows 薄包装，内部转发到 `release.py`

## 一次性初始化签名

直接执行：

```powershell
python .\release.py init-signing
```

或者：

```bat
release.bat init-signing
```

也可以一次性传参数：

```powershell
python .\release.py init-signing keystore/gym-app-release.jks gym-app yourStorePassword yourKeyPassword
```

执行完成后会生成：

- `keystore\xxx.jks`
- `keystore.properties`

## 打包签名版 APK

签名配置完成后，执行：

```powershell
python .\release.py build-release
```

如果签名配置有效，Gradle 会输出正式签名后的 release APK：

```text
app/build/outputs/apk/release/app-release.apk
```

如果没有签名配置，则只会得到未签名包：

```text
app/build/outputs/apk/release/app-release-unsigned.apk
```

## 正式发布更新

```powershell
python .\release.py publish 1.4 "新增功能A" "修复问题B"
```

注意：

- `publish` 负责版本号、提交、推送和部署流程
- 能否安装覆盖更新，取决于你本地构建时是否使用了和历史版本一致的 keystore

## 如果你已经装过旧版 App

### 情况 1：你还保留着旧版签名文件

最好，继续使用原来的 keystore。

只要保证：

- `storeFile` 指向原 keystore
- `keyAlias` 和密码也还是原来的

就可以继续覆盖更新。

### 情况 2：你没有旧版签名文件

那旧版 App 基本无法被直接覆盖更新。

你只能二选一：

1. 卸载旧 App，再安装新签名的版本
2. 找回旧 keystore

卸载会导致本地数据丢失，除非你的数据已经做了云同步或导出备份。

## 最重要的保管建议

- keystore 要做离线备份
- 至少备份两份
- 记住 `storePassword`、`keyAlias`、`keyPassword`
- 不要把 keystore 和密码提交到 Git 仓库
